# T-SDK

Framework-agnostic API transport layer for the T-Suite ecosystem.

**Available in:** [Français](./README.fr.md) · [Español](./README.es.md) · [Italiano](./README.it.md) · [Deutsch](./README.de.md) · [Português](./README.pt.md) · [中文](./README.zh.md)

## Overview

T-SDK is a plugin-based, strongly-typed HTTP transport layer. Instead of configuring HTTP methods and routes per resource, you install a **sub-SDK** (a plugin for your backend convention) and declare only the resource names you need. The plugin provides all the methods.

```ts
import { createLomkitSDK } from '@t-suite/t-sdk/lomkit'
import { ofetch } from 'ofetch'

const TSDK = createLomkitSDK({
    transport: ofetch,
    baseURL:   'https://api.example.com',
    resources: ['user', 'product', 'order'] as const,
})

await TSDK.user.search({ filters: [{ field: 'active', value: true }] })
await TSDK.user.mutate({ mutate: [{ operation: 'create', attributes: { name: 'Alice' } }] })
await TSDK.product.destroy({ key: 42 })
```

### Key principles

- **Plugin architecture** — the sub-SDK owns the methods, the driver and the types
- **Resource = a name** — you pass `'user'` and get all plugin methods inferred by TypeScript
- **Method = a file** — each method is a self-contained file that resolves its own HTTP request
- **Named by the frontend team** — `softDelete`, `archive`, `publish` map to whatever the backend expects
- **CLI scaffolding** — add sub-SDKs, resources and methods without touching any existing file

## Architecture

### Sub-SDKs

A sub-SDK is a self-contained folder inside `t-sdk/src/` that bundles a **plugin** (methods + types) for a specific backend convention.

```
t-sdk/src/
├── core/           ← createTSDK, TSDKError
├── lomkit/         ← Lomkit sub-SDK
│   ├── driver.ts
│   ├── methods/    ← search.ts, mutate.ts, destroy.ts, restore.ts, actions.ts
│   ├── resources/  ← per-project resource interfaces (generated by CLI)
│   ├── types/      ← ILomkitSearchPayload, ILomkitMutatePayload, …
│   └── index.ts    ← createLomkitSDK, lomkitPlugin
├── rest/           ← REST sub-SDK
│   ├── methods/    ← list.ts, get.ts, create.ts, update.ts, delete.ts
│   └── index.ts    ← createRestSDK, restPlugin
└── index.ts
```

### Method files

Each method is a file that exports an `IMethodDefinition` — a single `resolve` function that maps `(resource, payload) → IRequest`.

```ts
// lomkit/methods/destroy.ts
export const destroy: IMethodDefinition = {
    resolve: (resource, payload) => {
        const { key } = payload as { key: number | string }
        return {
            url:    `/${resource}/mutate`,
            method: 'POST',
            body:   { mutate: [{ operation: 'destroy', key }] },
        }
    },
}
```

The frontend name (`destroy`) and the backend payload are **decoupled**. Name the method whatever makes sense in your codebase.

## Built-in sub-SDKs

### Lomkit

For backends using [Laravel Lomkit](https://github.com/lomkit/laravel-rest-api).

```ts
import { createLomkitSDK } from '@t-suite/t-sdk/lomkit'

const TSDK = createLomkitSDK({
    transport: ofetch,
    baseURL:   'https://api.example.com',
    resources: ['user', 'product'] as const,
})
```

| Method | Endpoint | Description |
|---|---|---|
| `search(payload)` | `POST /{resource}/search` | Filter, sort, paginate |
| `mutate(payload)` | `POST /{resource}/mutate` | Create, update in batch |
| `destroy({ key })` | `POST /{resource}/mutate` | Destroy by key |
| `restore({ key })` | `POST /{resource}/restore` | Restore soft-deleted record |
| `actions({ action, ...body })` | `POST /{resource}/actions/{action}` | Custom backend actions |

#### Search example

```ts
const result = await TSDK.user.search({
    filters: [
        { field: 'active',      value: true },
        { field: 'role',        operator: 'in', value: ['admin', 'editor'] },
    ],
    sorts:   [{ field: 'created_at', direction: 'desc' }],
    page:    1,
    limit:   20,
})
```

#### Mutate example

```ts
await TSDK.user.mutate({
    mutate: [
        { operation: 'create', attributes: { name: 'Alice', email: 'alice@example.com' } },
        { operation: 'update', key: 5, attributes: { name: 'Bob' } },
    ],
})
```

### REST

For conventional RESTful backends.

```ts
import { createRestSDK } from '@t-suite/t-sdk/rest'

const TSDK = createRestSDK({
    transport: ofetch,
    baseURL:   'https://api.example.com',
    resources: ['user', 'product'] as const,
})
```

| Method | Endpoint | Description |
|---|---|---|
| `list()` | `GET /{resource}` | Get all |
| `get({ id })` | `GET /{resource}/{id}` | Get one |
| `create(body)` | `POST /{resource}` | Create |
| `update({ id, ...body })` | `PUT /{resource}/{id}` | Replace |
| `delete({ id })` | `DELETE /{resource}/{id}` | Delete |

## CLI

### Add a sub-SDK

Scaffolds a new sub-SDK folder and updates `t-sdk/src/index.ts`.

```bash
t-sdk add graphql
# → creates t-sdk/src/graphql/ (methods/, resources/, types/, index.ts)
# → updates t-sdk/src/index.ts
# → the lead dev implements driver.ts
```

### Add a resource

Generates a typed resource interface. If a backend URL is provided, it attempts introspection.

```bash
# Pre-filled template (dev completes the fields)
t-sdk add-resource user --sdk=lomkit

# With introspection (sends POST /user/search, maps the response shape)
t-sdk add-resource user --sdk=lomkit --url=https://api.example.com
```

Generated file:

```ts
// t-sdk/src/lomkit/resources/user.ts
export interface IUser {
    id:         number
    name:       string
    email:      string
    created_at: string
    updated_at: string
}
```

### Add a method

Scaffolds a new method file and updates the SDK's `methods/index.ts`.

```bash
t-sdk add-method softDelete --sdk=lomkit
# → creates methods/softDelete.ts with a template
# → updates methods/index.ts
# → lead dev implements the resolve function
```

## Error Handling

Every failed request surfaces as a `TSDKError`.

```ts
import { TSDKError } from '@t-suite/t-sdk'

try {
    await TSDK.user.mutate(payload)
} catch (error) {
    if (error instanceof TSDKError) {
        error.status   // HTTP status code
        error.message  // Human-readable message
        error.errors   // Record<string, string[]> — field-level errors
    }
}
```

## Framework Adapters

### React / Next.js

```tsx
// app/layout.tsx
import { TSDKProvider } from '@t-suite/t-sdk-react'
import { TSDK } from '@/lib/tsdk'

<TSDKProvider sdk={TSDK}>{children}</TSDKProvider>
```

```tsx
import { useTSDKContext, useQuery, useMutation } from '@t-suite/t-sdk-react'

const sdk = useTSDKContext()

const { data, isLoading } = useQuery(() => sdk.user.search({ page: 1 }))
const { mutate }          = useMutation((p) => sdk.user.mutate(p))
```

### Vue / Nuxt

```ts
// plugins/tsdk.ts
import { provideTSDK } from '@t-suite/t-sdk-vue'
import { TSDK } from '@/lib/tsdk'

export default defineNuxtPlugin(() => provideTSDK(TSDK))
```

```vue
<script setup lang="ts">
import { useTSDK, useQuery } from '@t-suite/t-sdk-vue'

const sdk = useTSDK()
const { data, isLoading } = useQuery(() => sdk.user.search({ page: 1 }))
</script>
```

### Astro

```ts
// lib/tsdk.ts
import { defineTSDK } from '@t-suite/t-sdk-astro'
import { lomkitPlugin } from '@t-suite/t-sdk/lomkit'
import { ofetch } from 'ofetch'

export const TSDK = defineTSDK({
    transport: ofetch,
    baseURL:   'https://api.example.com',
    plugin:    lomkitPlugin,
    resources: ['user', 'product'] as const,
})
```

```astro
---
import { TSDK } from '../lib/tsdk'
const users = await TSDK.user.search({ limit: 10 })
---
```

## Extending with a custom sub-SDK

```bash
t-sdk add mybackend
```

Then implement `driver.ts` and add methods:

```ts
// t-sdk/src/mybackend/methods/customMethod.ts
import type { IMethodDefinition } from '../../types/ITSDK'

export const customMethod: IMethodDefinition = {
    resolve: (resource, payload) => ({
        url:    `/${resource}/custom`,
        method: 'POST',
        body:   payload,
    }),
}
```

```bash
t-sdk add-method customMethod --sdk=mybackend
t-sdk add-resource user --sdk=mybackend
```
